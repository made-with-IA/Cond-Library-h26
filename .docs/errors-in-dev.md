# Development Errors & Resolutions

A record of real errors encountered while building Biblios and how each was resolved.

---

## 1. TypeScript duplicate export error in `api-zod`

**File:** `lib/api-zod/src/index.ts`

**Error:**
```
error TS2308: Module './generated/types' has already exported a member named 'X'.
Consider explicitly re-exporting to resolve the ambiguity.
```

**Cause:**  
`lib/api-zod/src/index.ts` was re-exporting from both `./generated/api` (Zod schemas) and `./generated/types` (TypeScript types generated by Orval). Both modules export identically-named symbols, which TypeScript treats as an ambiguous duplicate.

**Resolution:**  
Remove the re-export of `./generated/types` entirely. The `index.ts` should only export from `./generated/api` (Zod schemas). TypeScript types are already available directly from `@workspace/api-client-react` and do not need to be re-exported from the Zod library.

```ts
// lib/api-zod/src/index.ts — correct
export * from "./generated/api";
// do NOT add: export * from "./generated/types";
```

## 2. Missing `@types/node` in `integrations-gemini-ai`

**File:** `lib/integrations-gemini-ai/`

**Error:**
```
Cannot find name 'process'. Do you need to install type definitions for node?
  Try `npm i --save-dev @types/node`.
```

**Cause:**  
The Gemini AI integration package used `process.env` to read environment variables, but `@types/node` was not declared as a dependency in its `package.json`, so TypeScript had no knowledge of the Node.js global `process` object.

**Resolution:**  
Add `@types/node` as a dev dependency to the package:

```bash
pnpm --filter @workspace/integrations-gemini-ai add -D @types/node
```

## 3. JWT_SECRET type narrowing failure in `auth.ts`

**File:** `artifacts/api-server/src/middlewares/auth.ts`

**Error:**
```
Argument of type 'string | undefined' is not assignable to parameter of type 'string'.
```

**Cause:**  
`process.env.JWT_SECRET` has type `string | undefined`. Even after a `if (!JWT_SECRET) throw new Error(...)` guard, TypeScript did not narrow the type to `string` for the rest of the module because the variable was declared with `const JWT_SECRET = process.env.JWT_SECRET` — TypeScript's narrowing does not persist across module scope for `const` from `process.env`.

**Resolution:**  
Explicitly assert the type after the guard by assigning to a new typed constant:

```ts
const _jwtRaw = process.env.JWT_SECRET;
if (!_jwtRaw) {
  throw new Error("JWT_SECRET environment variable is required but was not provided.");
}
const JWT_SECRET: string = _jwtRaw; // narrowed: string, not string | undefined
```

## 4. Express 5 route parameter typing

**Files:** `artifacts/api-server/src/routes/*.ts`

**Error:**
```
Argument of type 'string | ParsedQs | string[] | ParsedQs[]' is not assignable
to parameter of type 'string'.
```

**Cause:**  
In Express 5, `req.params.id` has the type `string | ParsedQs | string[] | ParsedQs[]` rather than the plain `string` it was in Express 4. Passing it directly to `parseInt()` or similar functions causes a TypeScript error.

**Resolution:**  
Wrap every `req.params` access with `String()` before using it:

```ts
// Before (breaks in Express 5)
const id = parseInt(req.params.id);

// After
const id = parseInt(String(req.params.id));
```

## 5. `@google/genai` not included in esbuild externals

**File:** `artifacts/api-server/build.mjs`

**Error at runtime:**
```
Error: Cannot find module '@google/genai'
```

**Cause:**  
The esbuild config listed certain Node.js built-in modules and heavy dependencies as `external` so they would be required at runtime instead of bundled. `@google/genai` was not in that list, so esbuild tried to bundle it. The package uses syntax or assets that esbuild could not process correctly, causing the bundle to fail or the module to be missing at runtime.

**Resolution:**  
Add `@google/genai` to the `external` array in `build.mjs`:

```js
external: [
  // ... existing externals ...
  "@google/genai",
],
```

## 6. `JWT_SECRET` environment variable missing at runtime

**Workflow:** `artifacts/api-server: API Server`

**Error:**
```
Error: JWT_SECRET environment variable is required but was not provided.
```

**Cause:**  
The API server performs a startup guard that throws immediately if `JWT_SECRET` is not set in the environment. The Replit Secret named `JWT_SECRET` had not been created yet (or was deleted), so the process exited before serving any requests.

**Resolution:**  
Add `JWT_SECRET` to Replit Secrets (via the Secrets tab or the environment-secrets tool) with a strong random string. After saving, restart the API Server workflow. The secret is automatically injected as `process.env.JWT_SECRET` at runtime.

## 7. FIFO reservation queue position gaps after cancellation

**File:** `artifacts/api-server/src/routes/reservations.ts`

**Bug:**  
When a reservation was cancelled mid-queue, the remaining entries kept their original `position` values, leaving gaps (e.g. positions 1, 3, 4 after position 2 was cancelled). The position displayed in the Reader Portal was then incorrect.

**Resolution:**  
After any cancellation, run a normalization pass that re-sequences all remaining `waiting` reservations for that book in insertion order:

```ts
// After marking a reservation as cancelled:
const remaining = await db
  .select()
  .from(reservations)
  .where(
    and(
      eq(reservations.bookId, bookId),
      eq(reservations.status, "waiting")
    )
  )
  .orderBy(asc(reservations.createdAt));

for (let i = 0; i < remaining.length; i++) {
  await db
    .update(reservations)
    .set({ position: i + 1 })
    .where(eq(reservations.id, remaining[i].id));
}
```

Additionally, if the cancelled reservation was in `notified` status, promote the next `waiting` entry to `notified` and set its `notifiedAt` / `expiresAt` timestamps.

## 8. Seed data inconsistency — books with active loans not marked as `borrowed`

**File:** `scripts/src/seed.ts`

**Bug:**  
The seed script inserted books with `status: "available"` and then created loans against them without updating the book status. The frontend showed books as available even though they had active loans, which caused incorrect UI state and broken business-rule assertions in tests.

**Resolution:**  
After inserting each loan, update the referenced book's status to match the loan outcome:

```ts
// After inserting a loan with status "active" or "overdue":
await db
  .update(books)
  .set({ status: "borrowed" })
  .where(eq(books.id, loan.bookId));

// After inserting a reservation that is "waiting":
await db
  .update(books)
  .set({ status: "reserved" })
  .where(eq(books.id, reservation.bookId));
```

All seed data was audited so that every book's `status` column accurately reflects its active loan/reservation state.
