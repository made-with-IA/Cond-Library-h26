## 1. Visão Geral do Produto

### 1.1 Problema

Condomínios com biblioteca comunitária gerenciam seus acervos de forma manual — cadernos, planilhas ou aplicativos genéricos que não foram feitos para este contexto. Isso causa:

- Perda de controle sobre livros emprestados e devoluções em atraso
- Dificuldade dos moradores em saber quais livros estão disponíveis
- Ausência de fila de reservas organizada
- Falta de comunicação eficiente quando um livro aguardado fica disponível
- Nenhum histórico de empréstimos para tomada de decisões sobre o acervo

### 1.2 Solução

**Biblioteca do Condomínio** é um sistema web completo que digitaliza e automatiza a gestão da biblioteca comunitária. Oferece:

- **Painel administrativo completo** para síndicos e bibliotecários
- **Portal do leitor** para moradores consultarem e reservarem livros pelo smartphone
- **Controle de fila de espera** com notificação via WhatsApp
- **Relatórios** para acompanhar o desempenho da biblioteca
- **Busca inteligente** via IA para preencher dados dos livros automaticamente

### 1.3 Público-Alvo

- Condomínios residenciais com biblioteca comunitária
- Bibliotecas de pequeno porte em associações, clubes e escolas

## 2. Objetivos e Métricas

### 2.1 Objetivos do Produto

| Objetivo | Indicador de Sucesso |
|---|---|
| Eliminar perda de controle sobre empréstimos | 100% dos empréstimos registrados no sistema |
| Reduzir devoluções em atraso | Taxa de atraso < 15% em 3 meses de uso |
| Aumentar a utilização do acervo | Taxa de disponibilidade de livros > 70% |
| Facilitar acesso dos moradores | ≥ 50% dos leitores usando o portal digital |
| Comunicação eficiente de fila | Tempo médio de notificação < 24h após disponibilidade |

### 2.2 Fora do Escopo de Métricas

- Receita financeira (sistema gratuito para o condomínio)
- Comparação com outros condomínios

## 3. Personas e Stakeholders

### 3.1 Administrador da Biblioteca

**Quem é:** Síndico, sub-síndico, ou morador voluntário responsável pela biblioteca.

**Características:**
- Acesso ao painel administrativo com login/senha
- Gerencia todo o acervo, leitores e empréstimos
- Precisa de relatórios para prestar contas ao condomínio
- Pode ou não ter conhecimento técnico

**Dores:**
- Tempo gasto controlando empréstimos manualmente
- Dificuldade em localizar livros que nunca foram devolvidos
- Falta de visibilidade sobre quais livros são mais procurados

---

### 3.2 Leitor Cadastrado (Morador)

**Quem é:** Morador do condomínio com cadastro ativo na biblioteca.

**Características:**
- Acessa o portal pelo smartphone ou computador
- Pode não ter muita familiaridade com tecnologia
- Quer saber rapidamente se um livro está disponível
- Precisa de lembrete quando o livro que reservou fica disponível

**Dores:**
- Ter que ir pessoalmente à biblioteca para saber se um livro está disponível
- Não saber quando vai chegar sua vez na fila de espera
- Esquecer a data de devolução e acumular atraso

---

### 3.3 Visitante (Morador sem cadastro)

**Quem é:** Qualquer pessoa que acessa a página pública sem estar cadastrada.

**Características:**
- Pode ver o acervo disponível e os comunicados
- Não pode realizar empréstimos ou reservas sem cadastro
- Incentivado a se cadastrar pelo administrador

## 5. Casos de Uso

### UC-01: Registrar Novo Empréstimo

**Ator:** Administrador

**Pré-condições:**
- Admin autenticado
- Livro com status `available` ou `reserved`
- Leitor com status `active`

**Fluxo Principal:**
1. Admin acessa "Empréstimos" → "Registrar Empréstimo"
2. Seleciona o leitor na lista ou busca por nome
3. Seleciona o livro (somente disponíveis ou reservados)
4. Confirma a data de devolução (padrão: 15 dias)
5. Sistema registra o empréstimo e atualiza o status do livro para `borrowed`
6. Se o livro estava `reserved` para este leitor, a reserva é marcada como `fulfilled`

**Fluxo Alternativo — Livro Reservado Para Outro:**
- 5a. Sistema bloqueia e exibe erro: "Este livro está reservado para [Nome]. Somente [Nome] pode retirar este livro."

**Pós-condições:**
- Empréstimo criado com status `active`
- Livro com status `borrowed`
- Reserva do leitor (se existir) com status `fulfilled`

---

### UC-02: Registrar Devolução

**Ator:** Administrador

**Pré-condições:**
- Admin autenticado
- Empréstimo com status `active` ou `overdue`

**Fluxo Principal:**
1. Admin localiza o empréstimo na lista
2. Clica em "Devolver"
3. Sistema registra `returnDate` = agora, status = `returned`
4. Sistema verifica fila de reservas para o livro

**Fluxo A — Com fila de espera:**
- Sistema muda livro para `reserved`
- Primeira reserva com status `waiting` é promovida para `notified`
- `notifiedAt` = agora, `expiresAt` = agora + 3 dias

**Fluxo B — Sem fila de espera:**
- Sistema muda livro para `available`

**Pós-condições:**
- Empréstimo com status `returned` e `returnDate` preenchido
- Livro com status `available` ou `reserved`
- Próximo da fila notificado (se existir)

---

### UC-03: Gerenciar Fila de Reservas

**Ator:** Administrador

**Pré-condições:**
- Admin autenticado
- Reservas existentes para o livro

**Ações disponíveis:**

| Ação | Gatilho | Efeito |
|---|---|---|
| **Notificar** | Apenas para primeiro da fila com status `waiting` | Status → `notified`, define `notifiedAt` e `expiresAt` (+3 dias) |
| **Avançar (skip)** | Reserva `notified` não retirada | Cancela atual, normaliza posições, promove próximo `waiting` |
| **Cancelar** | Qualquer reserva `waiting` ou `notified` | Status → `cancelled`, normaliza posições; se era único na fila e livro estava `reserved`, livro volta a `available` |
| **Cumprir** | Reserva `notified` quando empréstimo é criado | Status → `fulfilled`, normaliza posições, promove próximo ou libera livro |

---

### UC-04: Consulta Pública por E-mail

**Ator:** Visitante (morador)

**Pré-condições:** Nenhuma (público, sem autenticação)

**Fluxo Principal:**
1. Visitante acessa `/leitor`
2. Digita o e-mail cadastrado
3. Sistema busca o leitor e auto-marca empréstimos vencidos como `overdue`
4. Exibe: dados do perfil, empréstimos em atraso, empréstimos ativos, reservas pendentes

**Fluxo Alternativo — E-mail não encontrado:**
- Sistema exibe: "Nenhum leitor encontrado com este e-mail. Verifique o e-mail cadastrado ou fale com a administração."

---

### UC-05: Exportar Relatório de Devoluções

**Ator:** Administrador

**Pré-condições:** Admin autenticado

**Fluxo Principal:**
1. Admin acessa "Relatórios" → "Devoluções"
2. Aplica filtros: período (dueDateFrom/dueDateTo), status, nome do leitor, título
3. Visualiza a tabela filtrada
4. Clica em "Exportar CSV"
5. Browser gera e faz download do arquivo CSV

**Dados do CSV:**
- Livro, Autor, Leitor, Telefone, Data do Empréstimo, Data de Devolução Prevista, Data de Devolução Real, Dias em Atraso, Status

---

### UC-06: Notificar Leitor via WhatsApp

**Ator:** Administrador

**Pré-condições:**
- Admin autenticado
- Leitor com número de telefone cadastrado (mínimo 10 dígitos)
- Reserva com status `notified` ou `waiting`

**Fluxo Principal:**
1. Admin localiza a reserva na fila de espera ou na aba "Ativas"
2. Clica no botão WhatsApp
3. Sistema registra `notifiedAt` = agora via PATCH `/api/reservations/:id/notify`
4. Abre WhatsApp Web com número preenchido e mensagem pré-formatada
5. Admin envia a mensagem manualmente pelo WhatsApp

**Mensagem padrão:**
```
Olá, [Nome]! O livro "[Título]" está disponível para retirada na biblioteca do condomínio. 
Você tem até [data] para retirar. Caso não retire neste prazo, sua reserva será cancelada.
```

### 6.2 Padrão de Projeto

**Backend (Express 5):**
- Arquitetura de rotas modulares por domínio (books, users, loans, reservations, reader, auth)
- Middleware de autenticação separado para admin (`authMiddleware`) e leitor (`readerAuthMiddleware`)
- Validação de entrada com Zod v4 gerado via `drizzle-zod`
- Respostas sempre tipadas via TypeScript

**Frontend (React + Vite):**
- Geração automática de hooks React Query a partir do OpenAPI spec via Orval
- `ReaderAuthContext` para estado global de autenticação do leitor
- Layouts separados: `PublicLayout`, `AdminLayout`, `ReaderLayout`
- Design system unificado: Tailwind CSS + shadcn/ui + Lucide Icons

**Monorepo (pnpm workspaces):**
- `lib/api-spec`: fonte de verdade (OpenAPI YAML)
- `lib/api-client-react`: hooks gerados (nunca editar manualmente)
- `lib/api-zod`: schemas Zod gerados (nunca editar manualmente)
- `lib/db`: schema Drizzle compartilhado entre api-server e scripts

### 6.3 Fluxo de Dados da API

```
OpenAPI YAML (lib/api-spec/openapi.yaml)
        │
        │ pnpm --filter @workspace/api-spec run codegen
        ▼
┌─────────────────────────────────────┐
│  lib/api-client-react (gerado)      │  → hooks React Query no frontend
│  lib/api-zod (gerado)               │  → schemas Zod no backend
└─────────────────────────────────────┘
```

## 7. Banco de Dados

### 7.1 Diagrama Relacional

```
admins
  id · name · email · passwordHash · createdAt · updatedAt

users
  id · name · email · phone · block · house · status · passwordHash · createdAt · updatedAt

books
  id · title · author · genre · isbn · publishedYear · imageUrl · description · status · createdAt · updatedAt

loans
  id · bookId (FK→books) · userId (FK→users) · status · loanDate · dueDate · returnDate · createdAt

library_notes
  id · title · content · type · active · createdAt · updatedAt

reservations
  id · bookId (FK→books) · userId (FK→users) · position · status · notifiedAt · expiresAt · createdAt · updatedAt
```

### 7.2 Tabelas Detalhadas

#### `admins`

| Coluna | Tipo | Restrições | Descrição |
|---|---|---|---|
| id | serial | PK | Identificador único |
| name | text | NOT NULL | Nome completo |
| email | text | NOT NULL, UNIQUE | E-mail de login |
| passwordHash | text | NOT NULL | Hash bcrypt da senha |
| createdAt | timestamp | NOT NULL, DEFAULT now() | Data de criação |
| updatedAt | timestamp | NOT NULL, DEFAULT now() | Última atualização |

---

#### `users`

| Coluna | Tipo | Restrições | Descrição |
|---|---|---|---|
| id | serial | PK | Identificador único |
| name | text | NOT NULL | Nome completo do morador |
| email | text | NOT NULL, UNIQUE | E-mail (usado para login e consulta pública) |
| phone | text | NULL | Telefone (usado para WhatsApp) |
| block | text | NULL | Bloco/torre do apartamento |
| house | text | NULL | Número do apartamento/casa |
| status | user_status | NOT NULL, DEFAULT 'pending' | Status do leitor |
| passwordHash | text | NULL | Hash bcrypt (NULL = sem acesso ao portal) |

#### `books`

| Coluna | Tipo | Restrições | Descrição |
|---|---|---|---|

## 10. Regras de Negócio

### 10.1 Empréstimos

**RN-EMP-01: Elegibilidade do livro**
- Livro deve estar com status `available` ou `reserved`
- Se `reserved`: somente o leitor que está como `notified` na fila pode retirar
- Status `draft`, `borrowed`, `lost`, `unavailable` bloqueiam empréstimo

**RN-EMP-02: Elegibilidade do leitor**
- Leitor deve ter status `active`
- Status `pending`, `inactive`, `blocked` bloqueiam empréstimo

**RN-EMP-03: Efeito do empréstimo no livro**
- Livro vai para `borrowed` imediatamente ao registrar empréstimo

**RN-EMP-04: Efeito do empréstimo na reserva**
- Se o leitor tinha reserva `notified` para o livro → reserva vai para `fulfilled`
- Posições da fila são normalizadas após `fulfilled`

**RN-EMP-05: Marcação de atraso**
- `GET /api/loans` auto-marca empréstimos com `dueDate < agora` e status `active` como `overdue`
- Esta atualização ocorre no momento da requisição (lazy evaluation)

**RN-EMP-06: Devolução e fila**
- Devolução com fila presente: livro → `reserved`, primeiro `waiting` → `notified` (com `notifiedAt` e `expiresAt`)
- Devolução sem fila: livro → `available`

---

### 10.2 Reservas e Fila

**RN-RES-01: Posição na fila**
- Posições são inteiros sequenciais começando em 1
- Ao adicionar: `position = MAX(position para o bookId) + 1`
- Ao cancelar/cumprir: posições dos registros seguintes são decrescidas em 1 (normalização FIFO)

**RN-RES-02: Reserva direta**
- Se livro está `available` ao adicionar reserva → livro vai imediatamente para `reserved` e reserva já nasce como `notified`
- Se livro está `borrowed` → reserva nasce como `waiting`

**RN-RES-03: Elegibilidade para reserva**
- Somente leitores com status `active` podem ser adicionados à fila

**RN-RES-04: Notificação e prazo**
- Ao notificar: `notifiedAt = agora`, `expiresAt = agora + 3 dias`
- Somente o primeiro `waiting` por livro pode ser notificado

**RN-RES-05: Expiração**
- Reservas `notified` após `expiresAt` são consideradas "expiradas"
- Expiração não ocorre automaticamente; admin deve usar "Avançar" para forçar skip
- Ao "Avançar": reserva atual → `cancelled`, próximo `waiting` → `notified`

**RN-RES-06: Cancelamento da única reserva**
- Se cancelar a única reserva de um livro `reserved` → livro volta para `available`

---

### 10.3 Acervo

**RN-ACE-01: Status do livro é calculado pelo sistema**
- Admin não altera o status do livro diretamente via empréstimo/devolução/reserva
- Status é sempre resultado das operações nas tabelas `loans` e `reservations`
- Admin pode alterar o status manualmente na edição do livro (para casos como `lost`, `unavailable`)

**RN-ACE-02: Exclusão de livro**
- Livro com registros na tabela `loans` não pode ser excluído (FK sem CASCADE)
- Admin deve inativar o livro (status `unavailable`) em vez de excluir

---

### 10.4 Autenticação

**RN-AUTH-01: Dois contextos independentes de JWT**
- Admin: token gerado em `POST /api/auth/login`, verificado por `authMiddleware`
- Leitor: token gerado em `POST /api/reader/auth/login`, verificado por `readerAuthMiddleware`
- Tokens não são intercambiáveis

**RN-AUTH-02: Senha do leitor é opcional**
- Leitor cadastrado pelo admin sem senha não tem acesso ao portal autenticado
- Leitor pode acessar `/leitor` (consulta pública por e-mail) sem senha
- Leitor pode definir/alterar senha pelo perfil após login

## 11. Autenticação e Segurança

### 11.1 Fluxo de Autenticação Admin

```
1. POST /api/auth/login { email, password }
2. Servidor: bcrypt.compare(password, admin.passwordHash)
3. Sucesso: jwt.sign({ adminId, email }, JWT_SECRET, { expiresIn: '7d' })
4. Cliente: armazena token em localStorage
5. Requisições: header Authorization: Bearer <token>
6. Middleware: jwt.verify(token, JWT_SECRET) → req.admin
```

### 11.2 Fluxo de Autenticação do Leitor

```
1. POST /api/reader/auth/login { email, password }
2. Servidor: bcrypt.compare(password, user.passwordHash)
3. Sucesso: jwt.sign({ userId, email }, JWT_SECRET, { expiresIn: '30d' })
4. Cliente: armazena token em localStorage
5. Requisições: header Authorization: Bearer <token>
6. Middleware: jwt.verify(token, JWT_SECRET) → req.reader
```

### 11.3 Considerações de Segurança

| Aspecto | Implementação |
|---|---|
| Senhas | Hash com bcrypt (salt rounds padrão) |
| JWT | HS256, secrets via env var `JWT_SECRET` |
| CORS | Configurado para aceitar origem do frontend |
| Entradas | Validadas com Zod antes de qualquer operação no DB |
| Erros | Mensagens genéricas expostas (sem stack traces em produção) |
| WhatsApp | Integração apenas no frontend (sem envio server-side), sem armazenamento de chaves |

## 12. Integrações Externas

### 12.1 Gemini AI (Google)

**Uso:** Busca automática de dados de livros durante o cadastro

**Endpoint:** `POST /api/gemini/book-search`

**Como funciona:**
1. Admin digita título, ISBN ou trecho no campo de busca na tela de novo livro
2. Backend envia a query para a API Gemini via cliente configurado com `AI_INTEGRATIONS_GEMINI_BASE_URL` e `AI_INTEGRATIONS_GEMINI_API_KEY`
3. Gemini retorna: título, autor, gênero, ano, ISBN, sinopse e URL de capa sugerida
4. Campos são preenchidos automaticamente no formulário; admin pode editar antes de salvar

**Configuração:** Via Replit AI Integrations (sem necessidade de chave própria no Replit)

---

### 12.2 WhatsApp

**Uso:** Notificação de disponibilidade de reserva

**Tipo de integração:** Link `wa.me/` gerado no frontend (sem API oficial do WhatsApp)

**Fluxo:**
1. Admin clica no botão WhatsApp em uma reserva ou empréstimo
2. Frontend chama `PATCH /api/reservations/:id/notify` para registrar `notifiedAt`
3. Frontend abre `https://wa.me/55{telefone}?text={mensagem_codificada}` em nova aba
4. Admin envia a mensagem manualmente pelo WhatsApp Web/app

**Limitações:**
- Admin precisa enviar a mensagem manualmente (não é envio automático)
- Requer que o leitor tenha WhatsApp instalado com o número cadastrado
- Validação mínima: número deve ter pelo menos 10 dígitos

| Relatórios de acervo mais acessado | Pode ser adicionado na próxima iteração |
| Integração com sistemas de condomínio (GestiCond, etc.) | Dependência de APIs de terceiros; avaliação futura |
| Multi-tenant (múltiplos condomínios) | Arquitetura atual é single-tenant |
